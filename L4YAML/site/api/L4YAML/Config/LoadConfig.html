    LoadConfig — bundled load-time configuration #

    Initiative 4 settled choice (Blueprint 08 §LoadConfig): bundle EqMode + DuplicateKeyPolicy into a single LoadConfig structure threaded through parse, compose, and construct.

    The default value ({}) gives spec-strict behaviour:

    • eqMode := .strict (error on cycle)
    • duplicateKeyPolicy := .error (libyaml default)

    This file holds types only; the parse/compose/construct threading lands in Phases 3–5 along with the indexed-type pipeline.

    Open subtype: EqMode.bisim witness #

    D3 settled Bisimulation typeclass as the witness shape; the typeclass itself lands in L4YAML/Algebra/Equivalence.lean (Phase 2 §3). Until then, EqMode.bisim carries no payload at the type level, and the parser refuses bisim mode if the typeclass is not in scope at the call site.

              inductive L4YAML.Config.DuplicateKeyPolicy :
              Type

              Policy for duplicate keys encountered during composition. Threaded through compose.

              • error : DuplicateKeyPolicy

                Default: parse error on duplicate (libyaml semantics).

              • first : DuplicateKeyPolicy

                Keep the first occurrence; silently drop later ones.

              • last : DuplicateKeyPolicy

                Keep the last occurrence (Python yaml default).

              • merge : DuplicateKeyPolicy

                Merge values via the supplied combinator. The argument is left abstract (α → α → α) at this layer; the concrete instantiation YamlValue → YamlValue → YamlValue lands when the indexed RepGraph API is wired up in Phase 4.
